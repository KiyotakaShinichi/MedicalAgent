# PHI & Privacy Limitations

**Read this before doing anything with real patient data.**

MedicalAgent is a portfolio engineering project. Its job is to demonstrate
safety-first applied AI and MLE patterns — *not* to handle real Protected
Health Information (PHI). This document spells out what the project is
**not** approved for, and what would be required before any real PHI
could be processed.

## What this project is not approved for

- **No real PHI.** Do not upload, paste, attach, or store any real patient
  record, real lab value, real imaging, real clinician note, real
  identifier, or any other data covered by HIPAA, GDPR, the EU AI Act, the
  Philippines Data Privacy Act, or any other applicable jurisdiction's
  health-data law.
- **No clinical deployment.** This codebase is not approved for any
  decision-support use in any hospital, clinic, telehealth service, or
  pharmacy. There has been no validation study, no clinician acceptance
  review, no IRB / ethics review, no regulatory clearance, and no
  Quality Management System.
- **No HIPAA / GDPR / PDPA / EU AI Act compliance.** The default
  configuration ships with no covered-entity controls, no Business
  Associate Agreement, no data processing agreement, no record-of-
  processing register, and no notification workflow.
- **No production privacy guarantees.** Synthetic / demo data ships
  intentionally for engineering practice. Anything you put into this
  system is treated as engineering data, not protected data.

## What the project uses today

- **Synthetic patient journeys** generated by the simulator in
  `Data/complete_synthetic_breast_journeys/`. All identifiers, names,
  labs, and imaging are simulator output, clearly labelled.
- **Demo accounts** (`P001`, `clinician`, `admin`) with toy passwords.
- **A local SQLite database** at `medical_agent.db` (configurable via
  `DATABASE_URL`).
- **Public research datasets** for imaging exploration only — see
  [DATA_CARD.md](../DATA_CARD.md) for source-by-source license terms.

If you see anything in a dashboard or chat that looks like a real
identifier, it is synthetic.

## What would be required before handling real PHI

This list is illustrative, not exhaustive. Each item below would need to
be designed, implemented, reviewed, audited, and maintained.

### Legal / compliance review
- Formal compliance review against the applicable health-data regime
  (HIPAA, GDPR + Article 9, EU AI Act high-risk classification, PDPA,
  PIPEDA, etc.).
- Data Processing Agreements (DPAs) with every party that touches PHI.
- Business Associate Agreements (BAAs) with covered-entity partners
  where HIPAA applies.
- Record of processing activities (Article 30 GDPR).
- Patient consent / authorization workflow and revocation handling.
- Clinical and ethical governance committee review (IRB / IEC).
- A device classification and (where applicable) regulatory clearance
  (FDA 510(k), MDR, IVDR, PMDA, TGA, MFDS — the right one depends on
  jurisdiction and intended use).

### Identity & access management
- Strict role-based access control (RBAC) and attribute-based access
  control (ABAC) for every patient-data endpoint.
- Phishing-resistant authentication (SSO + WebAuthn / TOTP), short
  session lifetimes, reauth on sensitive actions.
- Hardware-key admin access for any user that can read multi-patient
  data.
- Just-in-time access elevation with reviewer approval and an audit
  trail.
- Auto-revocation on role change / departure.

### Data protection
- Encryption in transit (TLS 1.3 with strict ciphers, HSTS).
- Encryption at rest for database, object storage, backups, and
  exported artifacts (KMS-managed keys, automatic key rotation).
- Field-level encryption / tokenization for direct identifiers where
  the workload allows it.
- Secrets management (no `.env` in source; use a vault / parameter
  store with KMS-encrypted entries).
- De-identification / anonymization pipeline for any analytics or model
  training use; safe-harbor or expert-determination per HIPAA §164.514.
- Re-identification risk assessment whenever de-identified data is
  joined.
- Data minimization at ingest and on every API response.

### Auditability
- Immutable, append-only audit log (write-once storage) for every
  PHI read, write, export, model inference, and admin action.
- Request-ID correlation between application logs, audit logs, and
  model trace logs.
- Tamper-evident hashing (Merkle chain or signed log entries).
- Independent review of audit-log access (admins of admins).
- Retention and deletion policy per jurisdiction (HIPAA 6 years +
  state additions; GDPR storage-limitation principle).
- Patient-initiated data export / deletion (Article 15 / 17 GDPR).

### Operational hardening
- Threat model (STRIDE / LINDDUN), refreshed on every architecture
  change.
- Privacy Impact Assessment (PIA) / Data Protection Impact Assessment
  (DPIA) per high-risk processing.
- Penetration testing (third-party).
- Vulnerability management with SLOs.
- Backup, restore, and disaster recovery exercised on a defined cadence
  with documented RPO / RTO.
- Incident response plan with notification timelines (72 h GDPR,
  HIPAA breach notification rule).
- Business continuity plan.

### Clinical / model governance
- Independent validation on real data with a defined performance
  threshold and clinician review.
- Continuous monitoring of calibration, subgroup performance, and
  drift on production data with predefined alerts.
- Human-in-the-loop guardrails on every decision-support output.
- Versioned model registry with reproducible training and signed
  artifacts.
- Rollback playbook with kill-switch.
- Reportable-incident workflow for unsafe model behavior.
- Clinician-facing transparency: model card, data card, safety card,
  uncertainty band per output.

### Organizational
- Named Data Protection Officer where the regime requires it.
- Workforce training on PHI handling, including engineers and AI
  reviewers.
- Vendor risk reviews for every third-party processor.

## What the project ships today (for honest comparison)

| Control | Status in this repo |
|---|---|
| Synthetic-data labelling | ✓ Every artifact carries a `data_source` / `limitations` block |
| Role boundaries | ✓ Patient / clinician / admin scoped via `get_*_access_context` deps |
| Cross-patient lookup block | ✓ Deterministic pattern in `security_guardrails.py` |
| Application audit log | ✓ `app_event_logs` table with `actor_role`, `route`, `request_id` |
| Encryption in transit | ✗ Dev HTTP only; deployment of TLS is the operator's responsibility |
| Encryption at rest | ✗ Local SQLite is unencrypted |
| Secrets management | ✗ `.env` file based; no vault |
| Immutable audit log | ✗ Audit table is append-only by convention, not by storage guarantee |
| Authentication | Demo-only credentials; not phishing-resistant |
| HIPAA / GDPR review | ✗ None |
| Regulatory clearance | ✗ None |

This list is not "we will add these features next week". This list is
**what real PHI handling would require**, listed honestly so that nobody
mistakes the current state for production-readiness.
