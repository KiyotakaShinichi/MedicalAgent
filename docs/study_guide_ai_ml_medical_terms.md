# Study Guide: AI/ML, Statistics, and Medical Terms

This guide explains the terms used in this project in plain language. The safest way to study this system is to remember its boundary:

> The project is a monitoring and clinician-review PoC. It does not diagnose, confirm response, or recommend treatment.

## Core ML Tasks

| Term | Plain meaning | How we use it |
|---|---|---|
| Classification | Predicts a category/class. | Predicts whether a synthetic patient journey has a favorable final outcome. |
| Binary classification | Classification with two classes. | `treatment_success_binary`: favorable vs not favorable synthetic outcome. |
| Regression | Predicts a number. | Predicts `response_score_percent`, a continuous MRI-response signal. |
| Hybrid model/signal | Combines multiple model outputs. | Combines classifier probability and regression response score. |
| Multimodal modeling | Uses multiple data types. | Uses CBC labs, symptoms, treatments, MRI-derived features, and notes/reports. |
| Longitudinal modeling | Models data across time. | Uses treatment-cycle rows across a patient journey. |
| Patient-level aggregation | Combines many cycle rows into one patient prediction. | Cycle probabilities are aggregated into patient-level evaluation. |
| Feature | Input variable to a model. | WBC, ANC, tumor size, stage, subtype, regimen, symptom severity. |
| Label/target | What the model learns to predict. | `treatment_success_binary`, toxicity labels, or `response_score_percent`. |
| Synthetic label | Label generated by simulator rules. | Useful for engineering practice, not real clinical truth. |

## Models and Algorithms

| Term | Plain meaning | How we use it |
|---|---|---|
| Logistic regression | Linear classifier that outputs probability. | Baseline classifier for synthetic and BreastDCEDL pCR tasks. |
| Random forest | Many decision trees averaged together. | Classifier and response regressor. |
| Extra trees | Similar to random forest, with more randomness. | Classifier and response regressor. |
| Gradient boosting | Builds trees sequentially to fix prior errors. | Current champion classifier. |
| SVM / RBF SVM | Finds a separating boundary, with nonlinear kernel. | Baseline classifier. |
| MLP | Small neural network over tabular features. | Baseline classifier. |
| Ridge regression | Linear regression with L2 regularization. | Baseline response regressor. |
| Huber regressor | Regression less sensitive to outliers. | Robust response-regression experiment. |
| Gradient boosting Huber | Boosted trees using robust Huber loss. | Robust response-regression experiment. |
| Median ensemble | Takes the median prediction across models. | Used for robust response-score uncertainty/ensemble behavior. |
| 1D CNN | Neural net that scans time-like sequences. | Temporal treatment-cycle baseline. |
| GRU | Recurrent neural network for sequences. | Temporal treatment-cycle baseline. |
| Small 2D CNN | CNN over image slices. | BreastDCEDL imaging baseline; weak and exploratory. |
| SHAP | Explains feature contribution to predictions. | Used as explanation direction for model outputs. |
| RAG | Retrieval augmented generation. | Retrieves knowledge-base chunks before LLM answers. |
| Hybrid search | Combines keyword and semantic retrieval. | Helps catch exact medical terms plus related meaning. |
| Reranking | Reorders retrieved chunks by relevance. | Improves context quality before answering. |
| Parent-child chunking | Retrieves small chunks but can include larger parent context. | Better context without losing precision. |
| Contextual compression | Removes noisy retrieved text. | Keeps answers shorter and more grounded. |
| Query rewriting / HyDE | Rephrases query for better retrieval. | Used conceptually for better RAG routing/retrieval. |

## Evaluation Metrics

| Term | Plain meaning | Good / bad intuition |
|---|---|---|
| Accuracy | Percent predictions correct. | Can mislead if classes are imbalanced. |
| Balanced accuracy | Average of sensitivity and specificity. | Better than accuracy when labels are imbalanced. |
| Precision | Of predicted positives, how many were correct. | Low precision means many false alarms. |
| Sensitivity / Recall | Of actual positives, how many were caught. | In medical monitoring, low sensitivity is risky. |
| Specificity | Of actual negatives, how many were correctly not flagged. | Low specificity means too many false alarms. |
| F1 score | Balance of precision and recall. | Useful single summary, but hides calibration. |
| AUROC | Ranking quality across thresholds. | 1.0 perfect, 0.5 random. |
| AUPRC / Average precision | Precision-recall quality under imbalance. | More useful than AUROC when positives are rare. |
| Brier score | Probability error. | Lower is better. |
| Calibration | Whether probabilities mean what they say. | A 0.70 probability should happen about 70% of the time. |
| ECE | Expected calibration error. | Lower is better; high ECE means unreliable probability language. |
| Confusion matrix | Counts TP, FP, TN, FN. | Best way to inspect what the model got wrong. |
| True positive | Model says positive, actual positive. | Correct catch. |
| False positive | Model says positive, actual negative. | False alarm / overoptimism. |
| True negative | Model says negative, actual negative. | Correct non-flag. |
| False negative | Model says negative, actual positive. | Missed case; important in medicine. |
| MAE | Average absolute error for regression. | Lower is better; easy to understand. |
| RMSE | Like MAE but punishes big errors more. | High RMSE means outliers are a problem. |
| R2 | How much variation regression explains. | Near 1 is strong, but can be inflated on synthetic data. |
| Bootstrap CI | Repeated resampling to estimate metric uncertainty. | Wide intervals mean the evaluation set is unstable. |
| Decision curve | Checks benefit of model threshold vs flag all/none. | Helps choose thresholds for review workflows. |
| Cost-sensitive evaluation | Gives FN and FP different costs. | In medicine, false negatives often cost more. |
| Subgroup performance | Metrics by group. | Checks if model is weak for subtype, stage, age, or regimen. |
| Drift | Data distribution changes over time. | Can break model reliability. |
| Grounding score | How much the answer matches retrieved context. | Higher is better for RAG. |
| Hallucination proxy | Heuristic estimate of unsupported claims. | Lower is better. |
| Source hit rate | Whether expected source was retrieved/cited. | Higher is better. |
| Attack block rate | Whether unsafe/prompt-injection cases are blocked. | Must be 100% for safety demos. |

## Data and MLOps Terms

| Term | Plain meaning | How we use it |
|---|---|---|
| Train split | Data used to fit model. | Development rows only after locked-holdout workflow. |
| Validation split | Data used to tune/check during development. | Internal synthetic split and calibration split. |
| Test split | Data used for final measurement. | Internal development test split in reports. |
| Locked holdout | Frozen data not used for tuning. | 120 synthetic patients scored once after training. |
| Leakage | Future/label info accidentally enters features. | Temporal leakage audit checks for this. |
| Dataset lineage | Record of where data came from. | Hashes, schema, seed, feature lineage. |
| Dataset hash | Fingerprint of dataset contents. | Proves exact data version. |
| Schema version | Version of column/data contract. | Prevents silent feature changes. |
| Feature store | Shared source of training/serving features. | Local manifest only, account-free PoC version. |
| Model registry | Tracks model version/artifacts/metrics. | Local registry rows in DB. |
| Experiment tracking | Records params, metrics, artifacts. | Local MLExperimentRun table. |
| Model card | Document for one model. | Intended use, limits, metrics, caveats. |
| System card | Document for the whole AI system. | Safety boundaries, risks, mitigations. |
| Error taxonomy | Named failure categories. | Makes mistakes reviewable instead of vague. |
| Claim boundary | What we are allowed to say. | Engineering PoC, not clinical validation. |

## How to Study This Project

1. Learn the patient timeline first: treatment cycle -> labs -> symptoms -> MRI -> intervention -> outcome label.
2. Learn the confusion matrix next: TP, FP, TN, FN.
3. Then learn AUROC/AUPRC/Brier/ECE.
4. Then learn why locked holdout and leakage audits matter.
5. Finally learn RAG safety: retrieve, cite, refuse, escalate.

The interview-ready framing:

> I built a safety-first clinical monitoring PoC. The ML predicts synthetic monitoring signals, the RAG answers grounded educational questions, deterministic rules handle urgent safety boundaries first, and clinician review remains the decision point.
